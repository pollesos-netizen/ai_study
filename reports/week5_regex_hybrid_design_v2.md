# 5주차 정규식 + AI 하이브리드 탐지 설계 결과 v2

## 1. 목적

4주차까지 진행한 AI 문장분류 모델은 성명, 건강정보, 복지정보, 계약정보 등 문맥형 민감정보를 탐지하는 데 활용 가능성을 보였다.

다만 이메일, 주민등록번호, 내부 IP, VLAN, 포트 등 형태가 명확한 패턴형 정보는 AI 모델보다 정규식 기반 탐지가 더 안정적이다.

5주차에서는 다음 구조를 설계하였다.

```text
정규식 탐지
+ AI 문장분류
+ 규칙 기반 C/S/O 등급 산정
= 하이브리드 개인정보/민감정보 탐지 구조
```

이후 중요한 방향이 추가되었다.

```text
문서 전체 등급 판단이 주목적이 아니라,
개인정보/민감정보가 탐지된 위치를 사용자에게 알려주는 것이 핵심 목적이다.
```

따라서 최종 목표는 다음과 같이 재정의한다.

```text
문서 파일
→ 문서 파서
→ 위치 정보를 가진 TextUnit 생성
→ 정규식 + AI 하이브리드 탐지
→ 탐지 위치, 탐지 항목, 등급, 조치 방식 반환
→ 필요 시 문서 전체 등급은 부가적으로 산정
```

## 2. 최종 구현 방향

### 기존 관점

```text
문장 입력
→ 정규식 탐지
→ AI 문장분류
→ 문장 단위 C/S/O 판단
```

### 수정된 관점

```text
문서 입력
→ 문서 유형별 파싱
→ 위치 정보 포함 TextUnit 생성
→ TextUnit별 정규식/AI 탐지
→ Detection 결과 생성
→ 사용자에게 탐지 위치 표시
→ 비식별화 시 locationMeta 기반으로 원문 수정
```

즉, 문서 전체 등급은 다음 단계의 부가 기능이다.

```text
탐지 위치 목록이 먼저
문서 전체 등급은 탐지 결과들의 최고 등급으로 나중에 계산
```

| 구분 | 역할 | 주요 대상 |
|---|---|---|

| TextUnit | 위치 정보가 보존된 분석 단위 | 슬라이드, 셀, 문단, 페이지 등 |

| 비식별화 엔진 | 탐지 위치와 메타데이터 기반 원문 수정 | 마스킹, 삭제, 치환, 범주화, 재작성 |

## 4. TextUnit 개념

문서 파서가 반환하는 `TextUnit`이 탐지 엔진이 처리할 분석용 청크이다.

다만 RAG에서 말하는 chunk와 혼동을 피하기 위해 이 프로젝트에서는 `TextUnit`이라는 이름을 사용한다.

```text
문서 파일
→ 문서 파서
→ TextUnit 목록
→ 정규식/AI 탐지
→ Detection 결과
```

### TextUnit 기본 구조

```js
{
  text: "...",
  locationLabel: "...",
  locationMeta: { ... }
}
```

| 필드 | 의미 |
|---|---|
| `text` | 정규식/AI 탐지 대상 텍스트 |
| `locationLabel` | 사용자에게 보여줄 위치 정보 |
| `locationMeta` | 프로그램이 비식별화할 때 사용할 내부 메타데이터 |

---

## 5. locationLabel과 locationMeta 분리

사용자에게 보여줄 정보와 프로그램 내부 처리용 정보를 분리한다.

### locationLabel

사람이 문서를 열고 찾아갈 수 있는 수준의 위치 정보이다.

예시:

```text
3번째 슬라이드
계약내역 탭 B12 셀
17번째 문단
5페이지
```

### locationMeta

프로그램이 나중에 비식별화하거나 원본 문서를 수정할 때 사용하는 내부 위치 정보이다.

예시:

```js
{
  fileType: "xlsx",
  sheetName: "계약내역",
  cellRef: "B12",
  row: 12,
  col: 2
}
```

사용자 화면에는 기본적으로 `locationLabel`만 표시하고, `locationMeta`는 숨겨서 보관한다.

## 6. 파일 형식별 위치 정보 정책

사용자에게 의미 있는 위치 정보만 우선 표시한다.

| 파일 형식 | 사용자 표시용 위치 | 내부 메타데이터 |
|---|---|---|
| PPTX | `3번째 슬라이드` | slideIndex, slideNo, 필요 시 shapeIndex, paragraphIndex |
| XLSX | `계약내역 탭 B12 셀` | sheetName, cellRef, row, col |
| HWPX | `17번째 문단` | sectionIndex, paragraphIndex, paragraphNo, xmlPath |
| DOCX | `17번째 문단` | paragraphIndex, paragraphNo, 필요 시 table 정보 |
| PDF | `5페이지` | pageIndex, pageNo, 필요 시 lineIndex 또는 bbox |
| HWP | `추출된 문단/구간` | 파싱 가능 수준에 따라 별도 관리 |

### 중요 판단

PPTX의 텍스트박스 번호, HWPX/DOCX의 표 번호, PDF 좌표값은 일반 사용자에게 의미가 낮다.

따라서 사용자 표시용으로는 우선순위를 낮춘다.

다만 향후 비식별화 자동 처리에는 필요할 수 있으므로 `locationMeta`에 숨겨서 보관할 수 있다.

## 7. XLSX 위치 보완 필요

기존 App.jsx의 XLSX 파싱은 시트명까지만 위치로 저장하는 구조였다.

```js
location: sheetName
```

이는 사용자에게는 부족하다.

수정 방향:

```js
locationLabel: `${sheetName} 탭 ${cellRef} 셀`
```

예시:

```text
계약내역 탭 B12 셀
```

내부 메타데이터:

```js
locationMeta: {
  fileType: "xlsx",
  sheetName: "계약내역",
  sheetIndex: 0,
  cellRef: "B12",
  row: 12,
  col: 2
}
```

XLSX는 사용자가 실제로 조치하려면 시트명과 셀 주소가 필요하므로, 다른 형식보다 위치 정밀도를 높여야 한다.

## 8. Detection 결과 구조

TextUnit을 분석해서 생성되는 탐지 결과는 다음 구조가 적절하다.

```js
{
  label: "이메일 주소",
  matched: "test@example.com",
  grade: "S",
  action: "마스킹",
  source: "regex",
  context: "담당자 이메일은 test@example.com입니다.",
  locationLabel: "계약내역 탭 B12 셀",
  locationMeta: {
    fileType: "xlsx",
    sheetName: "계약내역",
    cellRef: "B12",
    row: 12,
    col: 2
  },
  start: 8,
  end: 24
}
```

### 필드 의미

| 필드 | 의미 |
|---|---|
| `label` | 탐지 항목명 |
| `matched` | 실제 탐지 문자열 |
| `grade` | C/S/O 등급 |
| `action` | 권장 조치 |
| `source` | `regex` 또는 `ai` |
| `context` | 탐지 문맥 |
| `locationLabel` | 사용자 표시용 위치 |
| `locationMeta` | 내부 처리용 위치 |
| `start`, `end` | TextUnit.text 안에서의 위치 |

AI 탐지 결과는 특정 문자열이 없을 수 있으므로 `matched`는 빈 값 또는 문장 전체로 둘 수 있다.

## 10. 한국인 성명 추정 제외

한국인 성명은 정규식 탐지 대상에서 제외한다.

이유:

```text
1. 성명은 문맥 없이 판단하기 어렵다.
2. 정규식 기반 성명 추정은 오탐 가능성이 높다.
3. 성명은 향후 AI 문장분류 또는 NER 모델로 다루는 것이 적절하다.
```

예시:

```text
최지연 씨의 서류가 접수되었습니다.
```

이 문장은 정규식 탐지 결과는 없지만, AI 문장분류 모델이 개인정보로 판단할 수 있다.

## 11. Python 정규식의 \b 문제와 해결

기존 JSX 정규식에서는 `\b`를 사용해도 한글 조사와 붙은 값이 정상 탐지되었다.

하지만 Python 정규식에서는 한글이 Unicode 단어 문자로 처리되어 `\b`가 예상대로 동작하지 않을 수 있었다.

예시:

```text
900101-1234567이
test@example.com입니다
75.450MHz가
02:05:30부터
```

따라서 Python 구현에서는 `\b` 대신 숫자/영문 기준 경계 조건을 직접 사용하였다.

예시:

```python
r"(?<!\d)\d{6}[-–]\d{7}(?!\d)"
```

이 방식은 한글 조사와 붙은 값도 안정적으로 탐지한다.

향후 JSX나 다른 언어로 옮길 가능성이 있으므로, 정규식 자체보다 테스트 케이스를 공통으로 관리하는 것이 중요하다.

## 12. 구현 파일

```text
src/
  regex_detector.py
  hybrid_detector.py
```

### regex_detector.py

정규식 기반 패턴 탐지기이다.

주요 함수:

| 함수 | 역할 |
|---|---|
| `detect_patterns(text)` | 입력 문장에서 정규식 패턴을 탐지 |
| `get_max_grade(results)` | 탐지된 정규식 결과 중 최고 등급 반환 |
| `print_detection_result(text)` | 정규식 탐지 결과 출력 |

### hybrid_detector.py

정규식 탐지 결과와 AI 모델 예측 결과를 결합한다.

주요 함수:

| 함수 | 역할 |
|---|---|
| `load_ai_model(model_path)` | 4주차에서 저장한 AI 모델 불러오기 |
| `hybrid_classify(text, ai_model)` | 정규식 탐지 + AI 예측 + 최종 등급 산정 |
| `get_regex_grade_or_none(regex_results)` | 정규식 탐지 없음과 등급을 구분 |
| `decide_final_grade(regex_grade, ai_grade)` | TextUnit 단위 최종 등급 결정 |
| `select_highest_grade(grades)` | 문서 전체 등급 산정 시 사용할 최고 등급 선택 |
| `print_hybrid_result(result)` | 하이브리드 판단 결과 출력 |

## 13. 최종 등급 산정 원칙

TextUnit 하나에 대한 최종 등급은 다음 원칙으로 결정한다.

```text
1. 정규식 탐지 결과가 있으면 정규식 등급을 우선 적용한다.
2. 정규식 탐지 결과가 없을 때만 AI 문장분류 결과를 적용한다.
3. AI 예측은 최종 등급과 별개로 참고 근거로 함께 보관한다.
```

이유:

```text
정규식 탐지 대상은 예외가 거의 없는 명확한 패턴형 정보이므로,
AI보다 정규식 결과를 더 신뢰하는 것이 적절하다.
```

## 14. 정규식 탐지 없음과 O 등급 구분

중요한 수정 사항이다.

```text
정규식 탐지 없음 ≠ O 등급
```

| 상태 | 의미 |
|---|---|
| 정규식 기준 등급: 해당 없음 | 정규식 탐지 대상이 발견되지 않음 |
| 최종 C/S/O 등급: O | 정규식 탐지 없음 + AI도 일반으로 판단 |
| 최종 C/S/O 등급: S | 정규식 탐지 없음 + AI가 개인정보 또는 민감정보로 판단 |
| 최종 C/S/O 등급: C/S | 정규식 탐지 결과가 있어 해당 등급 우선 적용 |

핵심 로직:

```python
def get_regex_grade_or_none(regex_results):
    if not regex_results:
        return None

    return get_max_grade(regex_results)
```

```python
def decide_final_grade(regex_grade, ai_grade):
    if regex_grade is not None:
        return regex_grade, "정규식 탐지 결과 우선 적용"

    return ai_grade, "정규식 탐지 없음 — AI 문장분류 결과 적용"
```

## 16. 현재 남은 이슈

### AI 모델 오탐

다음 문장은 일반 업무 문장에 가까우나, AI가 민감정보로 예측하였다.

```text
회의 결과를 요약하여 공유드립니다.
```

현재 하이브리드 구조에서는 정규식 탐지가 없기 때문에 AI 판단을 적용하여 최종 S가 된다.

이는 코드 로직의 문제라기보다 AI 모델의 오탐 문제이다.

개선 방향:

```text
1. 일반 업무 공유 문장 데이터 추가
2. AI 예측 확률 확인
3. 확신도가 낮은 경우 '검토 필요' 상태로 분리
```

### 건강정보의 C급 상향 필요성

현재 AI가 `민감정보`로 예측하면 S로 변환한다.

그러나 가이드라인 기준으로 건강정보, 병력, 진단서 등 법령상 민감정보는 C급 후보로 관리해야 할 수 있다.

향후 개선 방향:

```text
AI 예측: 민감정보
+ 건강정보 키워드 또는 category 확인
→ C 후보 또는 C로 상향 검토
```

## 17. 문서 전체 등급 산정 방식

문서 전체 등급은 탐지 결과를 모두 모은 뒤 부가적으로 산정한다.

```python
document_grade = select_highest_grade([
    detection["final_cso_grade"]
    for detection in detections
])
```

즉, 우선순위는 다음과 같다.

```text
1순위: 어디에 무엇이 탐지됐는가
2순위: 해당 탐지 항목의 등급과 조치
3순위: 문서 전체 최고 등급
```
