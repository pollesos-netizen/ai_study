# 사내 문서 비식별화 도구 — 발표 슬라이드

> **발표 목적:** 생성형 AI 활용을 위한 사내 문서 비식별화 도구 개발 결과 공유  
> **대상:** 팀 내부 / 유관 부서  
> **작성:** 빅데이터팀 AI혁신TF단

---

## 슬라이드 1 — 표지

**제목:** 사내 문서 비식별화 도구 개발 결과

**부제:** 생성형 AI 안전 활용을 위한 개인정보·민감정보 자동 탐지 시스템

> 발표자 메모: 이 도구는 "업무 문서를 외부 AI에 입력하기 전에, 무엇을 지워야 하는지 자동으로 알려주는 시스템"입니다.

---

## 슬라이드 2 — 배경: 왜 만들었나

**제목:** 생성형 AI 활용, 지금 이대로 괜찮은가?

- 직원들이 업무 문서를 ChatGPT·Copilot 등 외부 AI에 그대로 붙여넣고 있음
- 문서 안에는 **개인정보(이름·연락처)**, **기밀 정보(IP·설비 구성)**, **민감 정보(계약 단가·인사 정보)** 가 섞여 있음
- 사내 보안 가이드라인: C/S/O 등급 분류 후 등급별 외부 AI 활용 기준 존재
- **문제:** 직원이 매번 수동으로 확인하고 지우는 건 현실적으로 불가능

> 핵심 메시지: "규정은 있지만, 실행을 도와주는 도구가 없었다"

## 슬라이드 3 — C/S/O 등급 체계

**제목:** 어떤 정보를 어떻게 처리해야 하나

| 등급 | 정의 | 외부 AI 활용 | 예시 |
|------|------|-------------|------|
| **C (기밀)** | 외부 유출 시 시스템 공격·운영 위협 | **불가** | IP 주소, 설비 구성도, 보안 계정, 보호계전 설정값 |
| **S (민감)** | 개인정보 또는 내부 업무 정보 | 비식별화 후 가능 | 이름, 전화번호, 계약 단가, 인사 정보 |
| **O (일반)** | 공개 수준의 일반 정보 | 자유롭게 가능 | 회의 일정, 일반 안내문, 공개 규정 |

> 핵심 메시지: "이 도구는 C/S를 자동으로 찾아내어 직원 부담을 줄인다"

## 슬라이드 4 — 도구 개요

**제목:** 비식별화 도구가 하는 일

```
[직원]
  파일 업로드 (xlsx / csv / docx / pptx / hwpx / pdf)
       ↓
[비식별화 도구]
  자동 탐지 → 민감정보 마스킹/삭제 → 결과 안내
       ↓
[직원]
  비식별화된 파일 다운로드 또는 수정 위치 확인
       ↓
[외부 AI]
  안전하게 활용
```

- **브라우저 기반** — 설치 없이 사내 서버 접속으로 바로 사용
- **6종 파일 형식** 지원: xlsx, csv, docx, pptx, hwpx, pdf
- **3가지 탐지 방식** 조합: 정규식 + NER + AI

## 슬라이드 5 — 탐지 방식 3종

**제목:** 세 가지 탐지 소스의 역할 분담

| 방식 | 대상 | 결과 | 특징 |
|------|------|------|------|
| **정규식 (regex)** | 이메일, 전화번호, IP, 사번, 신용카드, 운전면허번호 등 | 자동 마스킹·삭제 | 패턴이 명확해 오탐 거의 없음 |
| **NER 모델** | 한국인 성명 | 자동 마스킹 | KoELECTRA 기반, 신뢰도 0.8 이상만 처리 |
| **AI 분류 모델** | 패턴 없는 서술형 민감 문장 | 검수 후보로 제안 | 최종 판단은 사용자 — AI는 추천만 |

> 핵심 메시지: AI는 "정답을 내리는 것"이 아니라 "검수할 후보를 줄여주는 것"

## 슬라이드 6 — 지원 파일 형식

**제목:** 6종 파일 형식, 두 가지 처리 방식

**자동 적용 모드 (파일 직접 수정)**

| 형식 | 처리 방식 |
|------|-----------|
| **xlsx** | 셀 단위 탐지 → 마스킹/삭제 적용 → 비식별화 파일 다운로드 |
| **csv** | 행×열 단위 탐지 → 마스킹/삭제 적용 → 비식별화 파일 다운로드 |

**안내 모드 (수정 위치 가이드 제공)**

| 형식 | 처리 방식 |
|------|-----------|
| **docx** | 단락·표 셀 탐지 → 수정 위치와 권장 조치 안내 |
| **pptx** | 슬라이드·도형·표 탐지 → 안내 |
| **hwpx** | 한글 문서 XML 파싱 → 단락·표 탐지 → 안내 |
| **pdf** | 줄 단위 추출 → 탐지 → 안내 (직접 편집 어려움 별도 고지) |

## 슬라이드 8 — AI 모델 개발 과정

**제목:** 574건 학습 데이터로 만든 분류 모델

**데이터 출처 두 가지:**

| 출처 | 건수 | 특징 |
|------|------|------|
| 사내 매뉴얼 검수 데이터 (trainset) | 336건 | 1·2·7호선 6개 직렬 26개 매뉴얼에서 추출, 직접 라벨링 |
| 일반 업무 문서 샘플 (privacy_v5) | 238건 | 이메일·사번·건강정보 등 다양한 개인정보·민감정보 포함 |

**최종 데이터셋 (v6_merged, 574건):**

- C급 140건 / S급 237건 / O급 197건
- 클래스 불균형 보정: `class_weight='balanced'` 적용

**모델:** TF-IDF char_wb(2-4gram) + word(1-2gram) + 키워드 카운트 피처 → LinearSVC

**성능:** Accuracy 0.647, **C recall 0.67** (비식별화 도구 특성상 C 미탐 최소화 우선)

## 슬라이드 9 — 모델 설계 결정: 왜 AI가 자동 처리하지 않나

**제목:** AI는 "추천"만, 판단은 사람이

**이유 1 — 모델 성능의 현실적 한계**
- 574건 데이터, Accuracy 0.647
- S↔C 경계는 "공격 악용 가능 여부"라는 의미론적 판단 → 패턴으로 100% 해결 불가

**이유 2 — 오판의 비대칭성**
- C를 O로 잘못 분류 → 기밀이 AI에 그대로 입력될 수 있음 (치명적)
- O를 C로 잘못 분류 → 직원이 불필요하게 검수하게 됨 (번거로움)
- 두 오류의 영향이 다름 → 자동 삭제보다 "검수 후보 제안" 방식이 안전

**이유 3 — 가이드라인 원칙**
> "확실할 때만 O. 불확실하면 상위 등급 유지"  
> — 생성형 AI 활용을 위한 데이터 보안 가이드라인 v1.0

## 슬라이드 10 — 시스템 구조

**제목:** 사내 서버 독립 배포 구조

```
[직원 브라우저]
      ↑↓ HTTP
[FastAPI 서버 (사내 서버 :8000)]
  ├── HTML UI 서빙 (static/index.html)
  ├── POST /api/detect   → 탐지 + 비식별화
  ├── GET  /api/download → 결과 파일
  └── POST /api/feedback → 사용자 판단 저장
      ↓
  [탐지 엔진 6종 + AI 모델 + NER 모델]
      ↓
  [피드백 저장소 (feedback_data/)]
      → 모델 재학습 데이터로 누적
```

- 외부 인터넷 연결 없이 **사내망 독립 운영**
- PHP 시스템 연동 가능 (cURL 호출 방식)
- 서버 시작 시 모델 1회 로딩 → 첫 요청부터 빠른 응답

## 슬라이드 12 — 주요 구현 이슈 및 해결

**제목:** 개발 전 과정에서 해결한 핵심 문제들

**1. docx·hwpx·pdf는 왜 "자동 수정"이 아닌가 (13~16주차)**

- xlsx는 셀 단위라 자동 수정 가능
- docx·hwpx·pdf는 서식(하이퍼링크, 추적 변경, 표 구조)이 얽혀 있어 수정 시 **숨겨진 서식 손상** 발생
- 결정: xlsx·csv → 자동 수정 후 다운로드 / 나머지 → 수정 위치 안내만
- 효과: 사용자가 원본 파일을 직접 수정하므로 서식 손상 없음

**2. Keras 모델 첫 요청 10~15초 지연 (18~19주차)**

- 요청마다 모델을 새로 로드하는 구조 → 첫 요청 10~15초 소요
- 해결: FastAPI lifespan으로 서버 시작 시 1회 로드 후 전역 캐시 유지
- 효과: 첫 요청부터 ~1초

**3. Keras 모델 입력 형식 오류 (18주차)**

- `model.predict(["텍스트"])` → `Invalid dtype: str` 오류
- 원인: Keras가 Python 문자열 리스트 직접 입력 불가
- 해결: `model.predict(tf.constant(["텍스트"]))`
- 교훈: 모델 입출력 형식은 반드시 실제 파일로 검증 필요

**4. AI가 분류 기준 설명 문장을 C로 오탐 (19주차)**

- "C등급은 외부 공개 시 기관에 중대한 피해를 줄 수 있는 기밀 정보입니다" → C로 분류
- 원인: 학습 데이터에 "C등급", "비식별화" 키워드가 C급 문장에만 있었음
- 해결: O급 분류 설명 문장 29건을 학습 데이터에 추가
- 효과: 피드백 오탐 9건 전부 O로 정확히 예측

**5. regex와 AI의 같은 문장 중복 탐지 (19주차)**

- regex가 이미 이메일을 탐지한 문장에 AI도 S급으로 추가 탐지
- 원인: 두 탐지기가 독립적으로 동작
- 해결: regex 결과가 있는 단락/셀은 AI 호출 생략
- 효과: AI의 역할을 "regex가 못 잡는 문맥 기반 탐지"로 명확히 분리

**6. 한글 PDF 띄어쓰기 누락 (16주차)**

- pdfplumber 기본값(`x_tolerance=3`)은 영문 PDF 기준 → 한글 추출 시 단어 사이 공백 소멸
- "파일을업로드합니다" 처럼 붙어서 추출 → regex 매칭 실패
- 해결: `x_tolerance=1` 적용
- 교훈: 오픈소스 라이브러리의 기본값이 한국어 환경에 맞지 않을 수 있음

**7. "Step 1" 같은 짧은 영문 오탐 (21주차)**

- "Step 1"(6자) → 개인정보(S) 0.85 확신도로 탐지
- 원인: char n-gram 모델이 OOD 입력(한국어만 학습)에서도 높은 확신도를 출력하는 구조적 한계
- 해결: 10자 미만 또는 한글 비율 20% 미만 텍스트는 AI 호출 사전 차단

**8. openpyxl 버전별 병합 셀 처리 오류 (12주차)**

- `MergedCellRange.__contains__`가 튜플을 받지 않아 특정 버전에서 오류
- `min_col_letter` 속성 없음 오류도 함께 발생
- 해결: 행/열 범위를 직접 비교하는 방식으로 교체
- 교훈: 라이브러리 버전 의존성은 실제 환경에서만 발견됨

## 슬라이드 15 — 마무리

**제목:** 핵심 정리

1. **문제:** 외부 AI 활용 전 민감정보 수동 확인 → 현실적으로 불가능
2. **해결:** 6종 파일 형식 자동 탐지 + 비식별화 도구 (사내 독립 배포)
3. **방식:** 정규식(자동) + NER(자동) + AI(검수 제안) 3단계 파이프라인
4. **원칙:** AI는 추천만, 판단은 사람 — 오판의 비대칭성 고려
5. **성장:** 사용 피드백이 학습 데이터로 누적 → 지속 개선 구조

> "규정은 있었지만 실행 도구가 없었다.  
> 이제 도구가 생겼다."
