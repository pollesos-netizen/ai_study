## 0. 이 문서의 목적

직렬별 검사표준매뉴얼·장애처리매뉴얼 같은 사내 업무 문서에서 **서술형 업무 민감정보**(정규식으로 잡히지 않는 절차·구성·조작 정보)를 C/S/O로 분류하기 위한 기준과, 후보 추출 스크립트가 따라야 할 규칙을 정의한다.

이 문서는 후보 추출 스크립트(`extract_candidates.py`, 가칭)의 **명세서**다. taxonomy·키워드·매칭 규칙은 모두 스크립트 구현의 근거가 된다.

핵심 원칙:
- 전체 문장 수작업 라벨링을 하지 않는다. 후보를 추려 사람이 볼 양을 줄인다.
- 키워드 매칭은 "C급 확정"이 아니라 "검수 후보로 승격"일 뿐이다.
- 결정론적으로 잡히는 식별자(IP, 사번 등)는 regex가 담당하고 AI 골든셋에서 제외한다.
- AI(현재 sklearn)의 임무는 패턴이 없는 **서술형 민감 문장** 판단으로 좁힌다.

## 1. C/S/O 판정 기준

가이드라인(슬라이드 8·9)의 정의를 사내 업무 문서 맥락으로 구체화한다.

**핵심 원칙 — C와 S를 가르는 것은 "조작 절차냐 아니냐"가 아니라 "외부 노출 시 위험도"다.** 조작 절차여도 단순 작업 순서면 S, 공격에 악용될 수 있으면 C다. 검수자는 문장마다 아래 3개 질문을 위에서부터 던지고, 처음 "예"가 나오는 등급으로 판정한다.

### C급 — 기밀 (공격 악용 가능)

> 외부에 노출되면 **공격자가 시스템을 멈추거나, 우회하거나, 취약점을 알 수 있는** 절차·구성·조작 정보. 가이드라인 슬라이드 8: "사이버 공격에 직접 악용 가능한 기술 정보", "시스템 우회 방법 등 외부 유출 시 위험을 직접 키우는 상세 매뉴얼".

판정 질문(Q1): **"이 절차/정보를 공격자가 손에 넣으면, 시스템 정지·오동작·우회·침투에 실제로 써먹을 수 있는가?"** → 예면 C.

해당 예:
- 시스템·장비를 **무력화/우회**하는 조작: BYPASS 취급, 보호장치 해제, 인터록 해제, 강제 절체
- 시스템에 **직접 침투·제어**: 설정파일 로딩(CLI 명령), 원격 접속+계정, shutdown/reboot 상세 절차, 카드 리셋 상세 순서
- **보호계전 설정값/정정치**(트립값, 동작 임계값) — 알면 보호체계 무력화 가능
- **세부 연계 구조·우회 방안** — 시스템 간 연결을 끊거나 돌리는 방법
- 절체 **우선순위·조건**(Lockout〉강제〉자동〉수동 같은 제어 위계)

### S급 — 민감 (개인정보 / 업무 / 단순 작업 순서)

> 개인정보이거나, 계약·평가·인사 같은 업무상 민감 정보이거나, **조작 절차라도 단순 작업 순서**여서 공격 악용도는 낮지만 내부 운영 정보로 보호할 것. 가이드라인 슬라이드 9 + 사내 확장 기준.

판정 질문(Q2): **"이건 공격 도구라기보다, 사람·거래·정형 업무에 관한 내부 정보인가?"** → 예면 S.

해당 예:
- 개인정보·연락: 성명, 사번, 내선, 연락처, 비상연락망, 통화기록
- 업무 행정: 근무조 편성, 업무분장, 인수인계, 승인·결재
- 계약·평가: 단가, 원가, 입찰 평가, 협상 전략, 장애 원인분석 보고서
- **정형 작업 순서**: 규정대로 수행하는 계통 작업 순서(예: 연락수전 순서, 차단기 순차투입 순서) — 조작이지만 "공격 악용"보다 "내부 업무 정보" 성격

### O급 — 공개 (단순 점검·확인)

> 조작 방법도 민감정보도 아닌, 단순 점검·확인·상태 표시·측정 또는 일반 설명.

판정 질문(Q3): **"이건 그냥 '봤다/쟀다/정상이다' 수준인가?"** → 예면 O.

해당 예:
- 단순 확인/점검: "LED 표시상태 확인", "동작상태 확인", "상태변화 확인"
- 측정·기록: "온도 측정", "전압 확인", "MIS 전산입력"
- 문서 목적·목차·개정이력, 일반 용어·법령 설명

### C vs S 경계 (가장 헷갈리는 지점)

둘 다 조작 절차일 때 가르는 단 하나의 기준:

| | C급 | S급 |
|---|---|---|
| 핵심 | 공격 악용 가능 (정지·우회·취약점) | 단순 작업 순서 / 내부 업무 정보 |
| 예 | "UPS 수동 BYPASS 취급", "설정파일 CLI 로딩", "절체 우선순위" | "연락수전(임학→작전) 순차투입", "차단기 순차투입 순서" |
| 직관 | 시스템을 **깨는** 법 | 규정대로 **일하는** 법 |

> **BYPASS 취급**은 보호장치를 *무력화*하는 방법이라 C. **연락수전 순서**는 정전 시 *정해진 절차대로* 전원을 돌리는 작업 순서라 S.

### 경계 원칙

가이드라인의 "확실할 때만 O" 원칙을 따른다. **불확실하면 상위 등급을 유지**한다(O보다 S, S보다 C 쪽으로 보수적 판정). 단, 이는 최종 검수자의 판단 기준이며, 후보 추출 스크립트가 등급을 확정하지는 않는다.

> **주의 — 키워드 트랙 ≠ 최종 등급.** 3장의 C-track/S-track 키워드는 "후보를 뽑기 위한 그물"일 뿐, 최종 C/S/O는 위 위험도 질문으로 사람이 판정한다. 예컨대 "절체"가 C-track 키워드로 걸려도, 그게 단순 작업 순서면 최종 등급은 S다. 스크립트의 `suggestedGrade`에 끌려가지 않는다.

## 2. regex 담당 영역 vs AI 담당 영역

| 구분 | 담당 | 처리 경로 | AI 골든셋 |
|---|---|---|---|
| 형식이 명확한 식별자 | regex / NER | autoTargets (자동) | **제외** |
| 서술형 업무 민감 문장 | AI (sklearn) | reviewTargets (검수) | **포함** |

**분리 이유:** regex가 100% 잡을 수 있는 식별자를 AI 학습에 넣으면, AI가 맞히든 틀리든 regex가 이미 한 일이라 학습 신호만 흐려진다. AI의 가치는 패턴 없는 서술형 판단에 있다.

### 2.1 regex 담당 (AI 학습 제외 대상)

아래는 후보 추출 시 regex로 검출하되, AI 골든셋 후보에서는 제외한다(`regexOnly=true`인 경우).

```
IP
VLAN
포트
계정
비밀번호
ID
서버 주소
전화번호
이메일
사번
주민등록번호
카드번호
계좌번호
```

> **매뉴얼 실증 패턴:** 검사표준매뉴얼에서 식별자는 이미 부분 마스킹되어 있다(예: `20*.23*.9*.71`, `ID(ad***)`, `PW(m*****)`, `http://192.168.1.88`). regex는 마스킹된 형태(`\d+\*\.\d+\*`)와 비마스킹 형태(`\d+\.\d+\.\d+\.\d+`)를 **둘 다** 잡아야 한다.
>
> **⚠️ 26개 코퍼스 식별자 실측 (regex 우선순위 근거):** 전체 26개 매뉴얼에서 정규식으로 측정한 식별자 노출량 — 비마스킹 내부 IP 215개, 마스킹 IP(`20*.23*`) 54개, 지역번호 괄호식 전화(`032) 451-3310`) 16개, 휴대폰(`010-…`) 3개, 대표번호(`18XX-…`) 5개. 즉 식별자는 코퍼스 전반에 실재하며 regex가 1차로 전량 포착해야 한다.
> regex 전화번호 패턴은 휴대폰(`010-\d{3,4}-\d{4}`), 대표번호(`1\d{3}-\d{4}`), **지역번호+괄호식**(`0\d{1,2}\)\s?\d{3,4}-\d{4}`)과 범위 꼬리(`~14`)를 모두 포착해야 한다. IP는 비마스킹(`\d+\.\d+\.\d+\.\d+`)과 마스킹(`\d+\*\.\d+\*`) 양쪽 모두.
> IP 표·포트 표·결선표·전화번호 표·비상연락망은 식별자 밀집 구역으로 `regex_identifier` 다발이며 regex 테스트셋의 핵심 샘플이 된다.

> **주의 (중요):** 한 문장이 식별자와 서술형 민감 키워드를 **동시에** 포함하는 경우가 많다.
> 예: "장애 발생 시 IP 10.0.1.1 장비를 수동 전환한다."
> 이 문장은 regex(IP)에도 걸리고 AI 키워드(장애·수동 전환)에도 걸린다.
> 이때 `regexOnly=false`로 두어 **AI 후보로 살려둔다.** regex 식별자가 있다고 무조건 제외하면 정작 중요한 절차성 후보를 놓친다.
> `regexOnly=true`는 **식별자만 있고 서술형 키워드가 전혀 없는** 문장에만 적용한다.

## 3. AI 후보 키워드 (서술형 업무 민감정보)

키워드는 그룹으로 분류하며, 각 키워드가 어느 등급 트랙(C/S)에 속하는지 태깅한다. 이 트랙 정보는 `candidateReason`에 함께 저장한다.

> **표기 규칙 (중요):** 각 그룹은 두 부분으로 나뉜다.
> - **키워드 목록** (코드블록 안): 스크립트가 그대로 읽는 **순수 단어 목록**. 쉼표로 구분된 단어만 있다. 라벨·페이지번호·설명 없음.
> - **근거 주석** (`>` 인용문): 사람이 읽는 설명. 스크립트는 무시한다. 페이지 번호(p27 등)와 출처는 여기에만 적는다.

> **보강 근거 (26개 매뉴얼 코퍼스 전수 빈도 분석):** 키워드는 1·2·7호선 6개 직렬 매뉴얼 26개 파일을 빈도 분석하여 검증·보강했다. 괄호 안 숫자는 코퍼스 전체 출현 횟수(공백 정규화 후).
>
> **직렬별 성격:**
> - **신호:** 점검 행위(O급)가 다수. `선로전환기`(484), `쇄정`(67), `제어금지`(15) 등 신호 고유 조작 + 접속·계정·IP.
> - **통신:** 리셋/절체/교체 조치 + 식별자(IP·전화번호·내선) 밀집.
> - **전력:** 보호계전 정정치(최고위험), 급·단전 계통조작, SCADA 명령어, 비상연락 전화번호(`급전분소` 121).
> - **기계:** PSD(승강장안전문)·승강·소방·배수·자동제어. `BYPASS 취급`, `인터록해제`, `재개폐`, `자동제어`(82)·`중앙제어`(42) 등 고유 조작.
> - **정보통신/전자:** 검사표준 위주(대용량). 단순 점검·`MIS점검`·`전산입력` 등 O급 행위가 압도적 → 약한 키워드 보강의 핵심 근거.

### 3.1 C-track 키워드

**그룹 C1 — 장애·복구 조치**

키워드:
```
장애, 복구, 복구완료, 장애 발생, 복구 절차, 장애 처리, 조치 순서, 조치내용,
재기동, 재부팅, 리셋, 전원 리셋, 카드 리셋, 카드 재부팅, 프로세서 재기동, 원격리셋, 원격 재기동,
초기화, 정상화, 응급 조치, 교체, 예비품 교체, 카드 교체, 탈장, 전원차단, 전원투입,
재투입, 순차투입, 원격투입, 자동재투입, 투입, 개방, 트립, 차단기 투입, 차단기 개방
```
> 통신: 거의 모든 항목이 "리셋/재부팅/교체"로 끝남. 전력: 보호계전 분야 전체가 차단기 "투입/개방/재투입/순차투입". 기계: 전원차단(75)·전원투입. "투입/개방/트립"은 단독 빈도가 높아 공출현 검토 대상(3.3 참고).

**그룹 C2 — 운용·조작 절차**

키워드:
```
수동 전환, 강제 조작, 수동 조작, 현장 조작, 수동조작, 수동제어, 원격제어,
조작 순서, 운용 절차, 임시 운용, 비상 운용, 수동모드, 전환명령, 일괄제어, 일괄급전, 일괄단전,
선로전환, 선로전환기, 쇄정, 제어금지, 인터록해제, 인터록, BYPASS 취급, 로컬 BYPASS,
재개폐, 수동운전, 비상운전, 강제진로, 진로설정, 현장취급
```
> 신호 고유: 선로전환기 조작(484), 쇄정(67), 제어금지/Control Tag(15) — 열차 진로를 직접 제어하는 최고 민감 조작. 기계 고유: PSD 인터록해제, BYPASS 취급, 가동문 재개폐. 전력: 수동운전/비상운전. 운용자가 시스템을 능동 조작하는 절차들.

**그룹 C3 — 절체·급단전·계통조작**

키워드:
```

육안점검, 정밀점검, 일일점검, 특별점검, 상태확인, 여부확인, 현장확인, 지속감시,
전산입력, MIS점검, 기록, 경보확인, Event확인, Alarm확인
```
> 코퍼스 빈도 최상위가 전부 단순 점검·기록 행위: 육안점검(332)·정밀점검(229)·상태확인·전산입력·MIS점검(856, 검사결과 시스템 입력). "통보·출동"은 전력 거의 모든 항목("전기파트에 통보 및 출동요청"). 정보통신·전자 검사표준은 이런 O급 행위가 압도적이라, 이를 약한 키워드로 묶지 않으면 대용량 매뉴얼 거의 전체가 후보가 된다.
> "투입·개방·트립"은 C1/C3에도 있으나 단독 빈도가 매우 높아, 단독으로는 후보를 만들지 않되 강한 키워드와 공출현 시 승격되도록 양쪽에 둔다.

**공출현 규칙:** 약한 키워드는 단독으로 후보를 만들지 않는다. 같은 TextUnit 안에서 **약하지 않은 키워드와 공출현할 때만** 후보로 승격한다.
- "전면 LED 표시 상태 확인" → `확인`+`상태`+`표시`(모두 약함) → **후보 아님 (O급)**
- "수동모드로 전환 후 절체시험" → `수동모드`(C2)+`절체시험`(C3)+`시험`(약함) → **후보 승격 (C)**
- "SSH 접속 후 shutdown 명령어 입력" → `SSH`·`접속`·`shutdown`·`명령어`(C5) → **후보 승격 (C)**
- "전기파트에 통보 및 출동요청" → `통보`+`출동`(모두 약함) → **후보 아님 (O급)**
- "해당변전소 원격투입 후 차단기 재투입" → `원격투입`·`재투입`(C1)+`투입`(약함) → **후보 승격 (C)**
- 예: "SSH 접속 후 shutdown 명령어 입력" → `SSH`(C5)+`접속`(C5)+`명령어`(C5) → **후보 승격 (C 후보)**
- 예: "화면 전환" → `전환` 단독 → **후보 아님**
- 예: "장애 발생 시 수동 전환" → `장애`(C1) + `전환`(약함) 공출현 → **후보 승격**

### 3.4 키워드 매칭 정규화 규칙

- 매칭 전 텍스트와 키워드 양쪽에서 **공백을 제거**한 뒤 부분일치 검사한다.
  - `장애 발생 시` ↔ `장애발생시`, `장애 발생시` 모두 매칭되도록.
- 대소문자·전각/반각은 정규화한다(영문 키워드 대비).

## 4. O급 자동 제외 (후보에서 거름)

다음 패턴에 해당하는 TextUnit은 후보 추출에서 자동 제외하여 검수 양을 줄인다. 단, **C/S 키워드가 함께 출현하면 제외하지 않는다**(제외보다 후보 승격이 우선).

```
- 목차 / 차례
- 개정 이력 / 제 개정 이력 / 이력
- 문서 목적 / 적용 범위
- 용어 설명 / 용어의 정의
- 법령명 단독 언급 (조문 인용 없는 단순 언급)
- 머리말/꼬리말 반복 텍스트, 페이지 번호
```

> S-track의 `점검 결과`는 단순 "점검 결과 정상" 같은 O급 서술과 자주 겹친다.
> 키워드로는 후보로 올리되, 최종 등급은 검수자가 판정한다(스크립트가 O로 자동 제외하지 않는다).

## 5. 후보 추출 결과 CSV 스키마

스크립트는 학습 데이터가 아니라 **검수용 후보 CSV**를 출력한다. `userGrade`는 빈칸으로 두고 사람이 채운다.

| 컬럼 | 채우는 주체 | 설명 |
|---|---|---|
| `sourceFile` | 스크립트 | 원본 매뉴얼 파일명 |
| `unitId` | 스크립트 | 내용 기반 ID (`sourceFile` + `headingPath` + 문단 해시). 재실행 시에도 동일해야 병합 가능 |
| `text` | 스크립트 | TextUnit 본문 |
| `headingPath` | 스크립트 | 절/항 경로 (예: `3. 장애처리 > 3.2 복구 절차`) |
| `precedingText` | 스크립트 | 앞 문단 context |
| `candidateType` | 스크립트 | **리스트**. 복수 가능 (예: `["ai_procedural_sensitive", "regex_identifier"]`) |
| `candidateReason` | 스크립트 | 매칭 근거. 키워드 + 트랙 (예: `[{"kw":"수동 전환","track":"C","group":"C2"}]`) |
| `regexOnly` | 스크립트 | 식별자만 있고 서술형 키워드 없을 때만 `true` (= AI 후보 제외) |
| `suggestedGrade` | (보류) | **비워둔다.** 앵커링 방지 — 검수자가 스크립트 제안에 끌려가지 않도록. 필요 시 검수 UI에서 라벨 입력 후 노출 |
| `userGrade` | **사람** | C / S / O / 오탐 |
| `memo` | 사람 | 검수자 메모 |

### candidateType 값 정의

```
ai_procedural_sensitive  : C/S-track 서술형 키워드 매칭 (AI 골든셋 후보)
regex_identifier         : regex 식별자 매칭
```

한 문장이 둘 다 해당하면 리스트에 둘 다 넣는다. 이때 `regexOnly=false`.

## 6. 후보 추출 → 골든셋 흐름

```
매뉴얼 텍스트 추출 (kordoc)
  → 문단/표 행 단위 TextUnit 생성 (headingPath, precedingText 보존)
  → O급 자동 제외 필터 적용
  → 키워드/regex 매칭 (약한 키워드 공출현 규칙 적용)
  → candidateType / candidateReason / regexOnly 태깅
  → 후보 CSV 출력 (userGrade 빈칸)
  → 사람이 300~400건 검수 (C/S/O/오탐)
  → v6 골든셋 구성
```

### 골든셋 구성 규모 (week20 목표)

| 구분 | 목표 건수 |
|---|---|
| C급 | 100 |
| S급 | 100 |
| O급 | 100 |
| 헷갈리는 사례 | 100 |

무작위 샘플이 아니라 C 키워드 후보 / S 키워드 후보 / 명백한 O / AI가 틀린 사례 / 애매한 사례를 섞는다.

## 7. AI 모델 적용 정책 (변경 없음)

업무 민감정보는 오탐·누락 리스크가 크므로 AI가 C/S/O를 **자동 확정하지 않는다.**

```
regex / NER          → autoTargets (자동 처리)
AI 업무 민감정보 판단 → reviewTargets (사용자 검수 대상)
```

AI 결과 표시 형식:
```
AI 추정: C급 가능성
감지 이유: 장애처리 절차 / 수동 조작 순서 포함
사용자 선택: C / S / O / 오탐
```

자동 삭제·자동 마스킹은 하지 않는다.

## 8. 미해결 / 다음 결정 사항

- TextUnit 분할 단위 확정: 문단 단위 vs 절/항 단위 vs 표 행 단위의 우선순위와 병합 규칙
- `precedingText` 포함 범위: 직전 1문단인지, headingPath까지인지
- 약한 키워드 공출현 판정 범위: 같은 TextUnit 내인지, precedingText 포함인지
- 골든셋 라벨링 도구: CSV 직접 편집 vs 간이 검수 UI
- 모델 업그레이드 시점: sklearn 유지 → (데이터 축적 후) ko-sroberta 임베딩 + LogReg
- **설정치(C4) 처리 경계:** 보호계전 트립값(`ΔI 3,000A`, `180V/0.5초`, `Trip 110℃`)은 수치 자체는 regex로 추출 가능하나, 민감성은 "이 값이 보호/차단 임계값인가"라는 **맥락**에 달려 있다(가이드라인 p19 수치 규칙과 동일). 단순 측정 허용범위(예: UPS 입력전압 342~418VAC)는 O급, 보호계전 정정치는 C급. regex로 수치를 뽑되 등급은 AI/검수가 판단하는 **하이브리드**가 적절. 후보 추출 시 설정치 포함 TextUnit은 C4 키워드(`정정치`, `트립 설정` 등)와 함께 잡아 review로 보낸다.

### 26개 코퍼스 분석으로 도출된 TextUnit 분할 방향 (잠정 결론)

문서 유형마다 의미 단위가 다르므로, 단일 분할 규칙보다 **문서 구조에 따른 적응형 분할**이 필요하다.

- **검사표준매뉴얼 (항목형):** "설비명 + 검사주기 + 검사항목 + 관련기준 + 검사방법 + 참고사항"이 하나의 검사 카드를 이룬다. 검사방법의 개별 줄(LED 확인 등)은 단독으로는 O급이지만, "접속 절차+계정"처럼 한 카드 안에서 결합될 때 민감해진다. → **검사 카드(설비+주기) 단위**로 묶고 headingPath에 설비명을 보존.
- **장애조치매뉴얼 (절차형, 통신·전력 공통):** "장애명 + 장애내용/판별 + 조치내용 + 처리흐름"이 하나의 조치 블록. 흐름도의 "장애발생→확인→절체→복구완료"는 쪼개면 의미를 잃는다. → **장애 항목(01-01, 1번, 40번 등) 단위**로 통째로 묶는 것이 옳다. 개별 조치 줄로 쪼개면 안 됨.
- **전력관제 특수 구조:** "참고사항(설정치)" 블록과 "관련규정" 인용이 각 항목 끝에 붙는다. 설정치 블록은 C4 후보로, 관련규정 단독 인용은 O급(법령 단순 언급)으로 처리. 근무요령·업무분장 섹션은 S-track(내부 운영 구조).
- **기계관제 구조:** PSD/승강/소방/배수 각 항목이 "관련장치 + 일반사항 + 연관절차 + 흐름도(경보→확인→BYPASS취급→복구)"로 구성. 흐름도가 OCR/변환 과정에서 깨져 단어가 흩어지는 경향이 있어(예: "BYPASS 취급", "인터록해제"가 표 셀로 분해됨) **항목 단위로 넓게 묶어** 단어 흩어짐을 보완해야 한다.
- **정보통신·전자 검사표준 (대용량):** 2만 줄 규모. 대부분 단순 점검·MIS 전산입력(O급)이라 O급 자동 제외 + 약한 키워드 공출현 규칙이 검수량을 좌우한다. 이 직렬에서 약한 키워드 필터가 약하면 후보가 수천 건으로 폭증한다.
- **공통:** 표는 행 단위로 분해하되 표 제목(headingPath)을 각 행에 부착한다. IP 표·포트 표·결선표·전화번호 표는 `regex_identifier` 다발로 처리.
